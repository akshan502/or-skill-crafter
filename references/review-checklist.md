# ADK Skill 最终防错审查表 (Review Checklist)

在向用户输出最终生成的 Skill 文件代码前，AI 必须对照此清单自检：

- [ ] **命名极其严格**：YAML 的 `name` 是否**全部且仅为小写字母、数字和连字符(-)**？必须与文件夹名完全一致。
- [ ] **合法的元数据**：YAML 头部是否**绝对清除了** `triggers` 等不支持的字段？（仅允许使用 `name`, `description`, `user-invocable`, `metadata` 等）。
- [ ] **保持 SKILL.md 精简**：新技能的主提示词是否干净？是否有把复杂的代码规则或模板剥离并提示用户存入 `references/` 或 `assets/` 目录？
- [ ] **门控机制生效**：新技能的 Pipeline 中，是否设计了防止 AI 直接一口气跑到终点、强迫它“暂停并等待用户输入”的门控指令（Gate Conditions）？
- [ ] **适用/不适用场景明确**：是否清楚写出了什么时候该用、什么时候不该用？尤其是“不适用场景”是否具体、可判断、能真正阻止误用？
- [ ] **输入不足先补问**：是否明确要求在关键信息缺失时先提问，而不是自行脑补？
- [ ] **失败兜底存在**：是否定义了工具不可用、输入不足、外部依赖失败时的降级策略？
- [ ] **目录分层正确**：是否明确区分了 `references/`、`assets/`、`scripts/`、`evals/` 的角色？
- [ ] **运行时数据不回流**：是否明确禁止把学习日志、偏好、缓存、状态文件等运行时数据写回 skill 本体目录？
- [ ] **环境耦合受控**：是否避免了写死不存在的特定环境工具名、路径或命令？若有绑定，是否明确说明这是环境专属技能？
- [ ] **边界约束充分**：是否写清楚了不能做什么、不能默认做什么、需要用户确认的事项是什么？
- [ ] **避免职责过宽**：该技能是否聚焦于一个清晰问题，而不是边界模糊的“万能技能”？
- [ ] **按需读取资源**：是否体现了渐进式披露，不会在一开始无差别读取全部 `references/` 和 `assets/`？
