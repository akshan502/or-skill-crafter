# Google ADK 技能设计核心规范

当你在设计一个新技能时，必须将以下理念贯彻到生成的代码中：

## 1. 渐进式披露 (Progressive Disclosure) - 三层加载机制
严禁将所有的约束和模板写在单一的 `SKILL.md` 提示词中！
- **第一层 (常驻内存)**：YAML Frontmatter 中的 `name` 和 `description`。极其简短，用于被系统发现。
- **第二层 (触发加载)**：`SKILL.md` 正文。<5000词，仅作为路由中心，定义角色、Pipeline 和硬性检查点。
- **第三层 (按需挂载)**：长篇检查清单/文档必须写入 `references/`；输出格式/模板必须写入 `assets/`；可执行动作写入 `scripts/`；评测样例可写入 `evals/`。

## 2. 目录型技能的标准分层
- `SKILL.md`：只放角色定义、适用范围、Pipeline、边界规则、门控点。
- `references/`：放规则、知识、清单、方法说明、静态参考资料。
- `assets/`：放模板、静态资源、示例骨架、输出格式参考。
- `scripts/`：放自动化脚本。
- `evals/`：放评测样例（可选）。

### 关键铁律
- **运行时数据不得写回 skill 本体目录。**
- 如学习日志、用户偏好、缓存、状态文件、执行结果等，必须写入独立的 data/output 目录，而不是 `references/` 或 skill 根目录。

## 3. 核心设计模式组合
- **Tool Wrapper (工具封装)**：将长篇规则剥离进 `references`。
- **Generator (生成器)**：将输出模板放入 `assets`。
- **Reviewer (审查器)**：提供单独的 check-list 文件用于代码或结果打分。
- **Inversion (反转)**：强制 Agent 改变本能，在行动前先抛出问题向用户收集上下文。
- **Pipeline (流水线)**：严格划分 Step，在涉及高风险或易错环境时，使用 **[硬性检查点]** 截断自动流。

## 4. 必须包含的结构元素
一个可用的技能，至少应包含以下内容：
- 适用场景
- 不适用场景
- 输入不足时的补问机制
- 失败兜底策略
- 输出要求
- 边界约束

## 5. 避免生成“万能技能”
一个好的技能应优先解决**一个清晰、聚焦、可判断边界的问题**。

避免生成这类职责过宽的技能：
- 既做文档生成，又做分析，又做抓取，又做自动化执行
- 适用场景写得过大，几乎什么都能触发
- 不适用场景写得过空，无法帮助 Agent 停下

当用户需求过宽时，应优先：
- 缩小技能职责
- 拆成多个子技能
- 或明确主技能 / 子技能边界

## 6. 环境耦合控制
除非该技能明确是某环境专属技能，否则不要在模板中写死：
- 特定 MCP 名称
- 特定浏览器扩展工具名
- 特定本地路径
- 特定云平台命令

优先描述为：**使用当前环境可用的浏览、抓取、执行、存储能力**。

## 7. 门控优先于冲动执行
技能不应一口气跑到底。凡是以下情况，应设置 **[硬性检查点]**：
- 用户需求尚不明确
- 资源规划尚未确认
- 涉及外部写操作
- 涉及高风险执行或自动化脚本
- 涉及依赖安装、环境修改或目录覆盖
